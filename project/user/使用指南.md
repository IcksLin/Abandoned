# 工程使用指南

本工程主要由定时线程运行函数和主循环函数组成。

由于是飞裸机开发，所以目前市面上的主流两家厂商都不约而同地使用了定时线程来模拟定时中断。在这个工程中，我们将在定时线程中使用的函数定义在 `project/code/my_global.cpp`，在主线程中运行的函数定义在 `project/code/my_task_function.cpp` / `project/code/my_task_function.hpp` 中。程序运行期间二者平行执行。

为了避免各个线程或者任务函数读取数据打架，我们在 `my_global` 中定义了一系列全局变量。接下来先介绍 `my_global` 的职能。

## `my_global` 的职能

1. 收容定时线程函数以及一些全局需要使用到的宏。
2. 做数据缓冲区处理。

   整个工程运行时由上层软件和下层软件分工合作。上层软件负责进行算法的一些计算，比如视觉算法负责计算方向；下层软件负责从设备上读取数据；而 `my_global` 则承担了数据缓冲这一职责。

   我们可以在 `my_global` 中看到：

   ```cpp
   float right_speed = 0.0f;
   float left_speed  = 0.0f;
   ```

   这两个值负责存储当前速度值。在定时线程中，`hight_frequence_encoder_get_speed_handler` 函数里的测速函数会疯狂地读取编码器，并将速度写入这两个值。它只管更新速度，其他事情一概不管。位于上位的 LADRC 计算函数以及其他函数或者任务想知道当前速度了，就只要从这两个地址中读取一下就行。

3. 承载一些全局作用函数，比如 `car_init` 小车总初始化函数。

下面是一些承担类似缓冲作用的变量清单：

| 变量 | 作用 |
| --- | --- |
| `right_speed` / `left_speed` | 当前左右轮速度反馈，由编码器线程更新 |
| `target_speed_r` / `target_speed_l` | 当前左右轮目标速度，由上层任务或者控制逻辑写入 |
| `speed_to_pwm_r` / `speed_to_pwm_l` | 控制器计算出来的 PWM 中间结果 |
| `onto` | 图像处理计算出的最终方向偏差 |
| `onto_control` | 方向环输出给速度环的转向控制量 |
| `cur_angle_speed` | IMU 线程更新的当前角速度 |
| `gray_img_ptr` / `rgb_img_ptr` | 摄像头图像数据指针 |

接下来重要的说完了，我们这里来分 P 介绍一下当前 `code` 中各个文件的作用。

## 主循环与全局线程

### `project/code/my_task_function.hpp` / `project/code/my_task_function.cpp`

负责收容在主循环中运行的函数。我们习惯将主循环中需要无限死循环执行、但是对周期要求不严苛的函数定义在这里，然后直接挂载在菜单函数中。

你可以发现我们的总循迹函数 `tracking` 就定义在里面。这个函数涵盖了图像获取到处理的全流程，后期如果你需要添加绕行相关功能，也可以直接在里面加。另外说一声，这个函数虽然直接塞在主循环中就可以使用了，但是由于没有加停止等功能，所以只会一直跑下去。

常用搭配：

- 搭配 `my_menu`：把 `tracking` 或其他测试函数挂到菜单叶子节点上。
- 搭配 `imgproc`：`tracking` 中真正执行视觉处理的是 `image_proc`。
- 搭配 `my_global`：读取摄像头、全局控制量和线程缓冲变量。

### `project/code/my_global.hpp` / `project/code/my_global.cpp`

这是全局对象和线程回调的大本营。电机、编码器、屏幕、按键、菜单、摄像头、PID、LADRC、IMU、惯性导航组件，基本都在这里进行统一声明和定义。

需要注意的是，这个文件不是普通工具库，它更像一个工程总线。你在这里加变量很方便，但是也很容易把线程之间的数据关系搞乱。所以新增全局变量时建议先想清楚：它到底是底层更新、上层读取，还是上层写入、底层执行。

常用搭配：

- 搭配 `motor`：编码器测速线程和电机输出线程都要调用电机模块。
- 搭配 `config_setting`：`car_init` 会从配置文件加载控制器参数。
- 搭配 `IMU963R` / `MadgwickAHRS`：按键扫描线程里顺手更新 IMU 和姿态解算。
- 搭配 `imgproc`：视觉方向 `onto` 会被控制线程读取。

## 菜单与按键

### `project/code/my_menu.hpp` / `project/code/my_menu.cpp`

这个文件是整个工程的菜单文件。你需要挂载菜单，就需要去 `Menu MyMenu::menu_table[]` 中按照格式将菜单设置好。

需要注意的是，只有叶子节点的菜单回调函数才会被执行，所以非叶子节点我们回调函数一般挂一个空指针。在挂载了菜单之后还不够，你需要去设置回调函数。老版本的菜单回调函数命名采用 `menu_mode_id` 规范以便于维护，但是新的却没必要，你可以直接根据当前菜单函数功能进行命名。

但是回调函数还是需要根据模板来写的，不然按键映射会出现问题。里面提供了模板函数 `menu_mode_1`。可用的菜单动作值可以去看菜单文件的下位程序文件 `my_key`，当前里面支持了短按和长按功能，一共七个有效值。

这里提醒一下，一定要实现返回功能，也就是在返回按键动作中将：

```cpp
g_menu_instance->mode_inter_flag
```

反转为 `0`。这个值是整个菜单功能从功能页面切换到菜单选择页面的关键标志位。你要是不先实现这个功能，你的菜单按键就成了一次性的，也就是说按进功能页面去了就直接出不来了。

菜单这里还有一个字体模式需要单独说清楚，因为它不是只在菜单文件里决定的，而是由 `project/user/CMakeLists.txt`、`libraries/zf_device/zf_device_ips200_fb.hpp` 和 `my_menu` 三个地方配合完成。

在 `project/user/CMakeLists.txt` 里面有这一句：

```cmake
option(USE_TTF "Whether to use Vector Fonts" ON)
```

它就是动态字体和静态字体的总开关。`ON` 表示启用 TTF 矢量字库，`OFF` 表示使用逐飞屏幕库原本那套 8x16 静态 ASCII 字库。这里注意一下，这个开关是在编译期生效的，不是程序运行时按一下菜单就能切换。

当 `USE_TTF` 为 `ON` 时，CMake 会做两件事：

1. 添加编译宏 `WHETHER_USE_TTF=1`。
2. 把 `assets/fonts/MiSans_Mini.ttf` 通过 `xxd -i` 转成 `font_data.cpp`，里面会生成 `vector_font_data[]` 字体数组。

当 `USE_TTF` 为 `OFF` 时，CMake 只会添加：

```cmake
add_definitions(-DWHETHER_USE_TTF=0)
```

也就是说不会生成 TTF 字体数据，屏幕库和菜单都会走静态字库分支。

然后看 `libraries/zf_device/zf_device_ips200_fb.hpp`。这个文件里会先判断 `WHETHER_USE_TTF`，如果工程没有定义它，就默认当作 `0`。当它为 `1` 时，屏幕类会声明：

```cpp
extern unsigned char vector_font_data[];
```

并启用基于 TTF 的 `print()` 绘制能力。这个 `print()` 可以显示 UTF-8 汉字，也可以按字号绘制，比如菜单里用的 `MENU_FONT_SIZE`。当它为 `0` 时，主要就靠 `show_string()` 这类静态字库接口，优点是轻，缺点是基本只能稳妥显示 ASCII，菜单名里不要随便塞中文。

最后才轮到 `my_menu`。菜单里面的 `draw_menu()` 会根据同一个宏切换：

```cpp
#if WHETHER_USE_TTF
    draw_menu_ttf(selected_menu);
#else
    draw_menu_static(selected_menu);
#endif
```

所以这套逻辑可以理解成：`CMakeLists.txt` 决定有没有 TTF，`zf_device_ips200_fb` 提供对应的底层绘字能力，`my_menu` 根据宏选择菜单绘制方式。

移植或者改菜单时需要记住几件事：

- 想显示中文菜单名，优先保持 `USE_TTF` 为 `ON`，并确认字体文件路径存在。
- 想减小体积或者避免生成字体文件，就把 `USE_TTF` 关掉，但是菜单名尽量保持 ASCII。
- `TTF` 模式走 `ips_display->print()`，静态模式走 `ips_display->show_string()`，所以功能页面里如果你手写显示代码，也要知道自己当前到底是哪一种字体模式。
- `ITEM_H` 和 `MENU_FONT_SIZE` 是 TTF 菜单布局的关键参数。字号调大后，行高不跟着调，文字就有可能上下撞在一起。
- 静态字库模式下使用的是固定字符宽高，菜单名称过长时会被截断，所以名字别写太长。

### `project/code/my_key.hpp` / `project/code/my_key.cpp`

这是菜单的下位按键扫描模块，负责把四个实体按键转换为菜单可以理解的动作值。

它内部做了一个小状态机：松开、消抖、按下、长按。短按是在松手时触发，长按也是松手后触发，这样可以避免长按过程中一直重复报事件。菜单系统不直接读 GPIO，而是通过 `get_menu_action` 获取 `MENU_UP`、`MENU_DOWN`、`MENU_OK`、`MENU_BACK` 这些动作。

常用搭配：

- 搭配 `my_global`：`key_scan_handler` 定时调用 `key_manager.scan_keys()`。
- 搭配 `my_menu`：菜单主循环调用 `get_menu_action()` 获取动作。

## 控制与电机

### `project/code/motor.hpp` / `project/code/motor.cpp`

这是电机和编码器的底层接口。它主要干三件事：初始化电机 GPIO/PWM、把速度指令映射成 PWM、从编码器读出当前速度。

这里面的参数非常依赖实际车模，比如 PWM 最大值、速度到 PWM 的换算系数、编码器方向、左右电机是否反接等。移植的时候不要直接相信这些数，尤其是 `MOTOR_PWM_MAX`、`PWM_PER_SPEED_UNIT`、`REMAP_WEIGHT` 这一类参数，最好重新标定一遍。

常用搭配：

- 搭配 `my_global`：线程回调里调用 `get_and_remap_speed`、`lardc_get_speed`、`motor_set_speed_ladrc`。
- 搭配 `my_pid`：PID 或 LADRC 算出结果后，最后都要落到这个文件的电机输出接口。

### `project/code/my_pid.hpp` / `project/code/my_pid.cpp`

这里面有两个主要控制器：`MyPID` 和 `PDController`，后面还包了一层 `SimpleMotorLADRC`。

`MyPID` 主要用于轮速环或者一些普通闭环控制，里面有积分限幅、死区、微分滤波、前馈、反向控制这些功能。`PDController` 更偏向方向控制，用于把图像方向偏差转换成转向修正量。`SimpleMotorLADRC` 则是把 `LADRC` 再包一层，让它更像电机速度控制器，直接输入目标速度和实际速度，输出 PWM。

常用搭配：

- 搭配 `config_setting`：控制器参数从 `car_config.txt` 或 VOFA+ 调参命令中写入。
- 搭配 `my_global`：全局的 `pid_r`、`pid_l`、`angle_speed_pd`、`onto_control_pd`、`ladrc_left`、`ladrc_right` 都在这里定义类型。
- 搭配 `LADRC`：`SimpleMotorLADRC` 内部真正调用的是 `LADRC::update`。

### `project/code/LADRC.hpp` / `project/code/LADRC.cpp`

这是线性自抗扰控制器。当前工程实际使用的是一阶 LADRC 方案，主要服务于速度环。里面包含跟踪微分器、扩展状态观测器和线性反馈律。

需要注意的是，`h` 一定要和真实调用周期对应。你如果线程周期改了，但是这里的 `h` 不改，控制器看起来就会非常玄学。`b0` 也是强相关实际系统的参数，换车、换电机、换减速比，都需要重新整定。

常用搭配：

- 搭配 `my_pid`：通过 `SimpleMotorLADRC` 作为速度控制器使用。
- 搭配 `config_setting`：LADRC 参数支持从配置文件加载和上位机更新。

## 姿态与惯性导航

### `project/code/MadgwickAHRS.cpp` / `project/code/IMU963R.hpp`

这两个函数是难兄难弟。其中 `MadgwickAHRS` 是上层程序，`IMU963R` 是下层程序。`MadgwickAHRS` 从下层程序读取值并进行解算，具体怎么调用的可以去看 `my_global` 中的 `key_scan_handler` 函数。

当前工程中为了避免过多的线程开辟，我将按键扫描、姿态获取和解算合并了。注意 `scan` 函数中的 `ahrs` 为解算器，隶属于 `MadgwickAHRS`，其传参顺序取决于你现实中陀螺仪怎么放置的。你需要自己依据你们的摆放方式来调整传入顺序和符号。

其中 `MadgwickAHRS` 提供了六轴的解算方法 `updateIMU` 和九轴的解算方法 `update`，请依照需要自行使用。额外需要注意的是周期，在解算器初始化的时候需要传入一个参数周期。我们工程中的是：

```cpp
MadgwickAHRS ahrs(100);
```

其中 `100` 是我们的解算器运行周期，单位是 Hz。这个一定要准确，不然 yaw 角跑飞就很有可能是它干的。

此外，零漂统计函数 `imu_device_type_enum IMUHandler::init(void)` 里面有一个：

```cpp
calibrate_offsets(100);
```

这个括号里面的值是进行多少次统计。此函数在初始化时调用，负责计算当前硬件条件下的零漂值。你设置得越大，它计算的周期越长，就越稳定。当然前提是它计算时小车需要保持绝对静止状态。这里还有一些运气成分，同时这个值有边缘效应，不必设置得特别大。

### `project/code/navigation.hpp` / `project/code/navigation.cpp`

这是惯性导航和路径复现模块。`Odometer` 负责把编码器脉冲变成累计里程，`PathTracker` 负责在录制时把里程和 yaw 角积分成路径点，在复现时从地图里找最近点和预瞄点。

这个模块目前更像一个路线记录与复现的基础架子。它依赖编码器线程持续更新左右轮里程，也依赖 `MadgwickAHRS` 给出 yaw。路径录制出来后，可以配合 `akima` 做平滑插值，再保存成地图文件。

常用搭配：

- 搭配 `my_global`：全局对象 `path_tracker_component` 在编码器线程里更新里程计。
- 搭配 `IMU963R` / `MadgwickAHRS`：需要 yaw 角参与位置积分。
- 搭配 `akima`：将录制路径平滑后输出为可复现地图。

### `project/code/akima.hpp` / `project/code/akima.cpp`

这是 Akima 插值模块，主要用来把录制出来的离散路径点变成更平滑的路径。它比普通线性插值顺滑，也比一些全局样条更不容易因为局部异常点把整段路带歪。

当前它提供了三类功能：计算插值系数、按指定里程评估坐标、把等距采样后的轨迹保存成地图文件。此外还提供了 `convert_txt_to_bin`，可以把文本地图转换成二进制地图，方便 `PathTracker` 直接加载。

常用搭配：

- 搭配 `navigation`：路径录制之后做平滑和地图生成。
- 搭配地图文件：文本格式一般是 `索引 x y`，二进制格式直接存 `MapPoint`。

## 图像处理与模型推理

### `project/code/imgproc.hpp` / `project/code/imgproc.cpp`

然后是图像处理，这是重头戏。我们将这个工程分为原子层和元素层。原子层就是移植的 17 年上交的开源方案，相当于 OpenCV 库中的一些基本函数，你可以进行组合使用。

在本工程中，我们通过原子层的功能函数将赛道左右边线的数组提取了出来，并计算左右最大弯曲度，然后通过左右弯曲度及其索引来判断元素类型并做处理，具体见 `element_status()`。

在 `element_status()` 中使用到了一个元素状态机的类 `tracking_decision_machine`，其中有一个 `state`，表明当前状态，目前占用 0、1、2、3、4 几个标志位。移植使用的话可以直接追加 5、6、7、8 并赋予意义。同时 `tracking_decision_machine` 里面还有负责记录当前最终中线是哪一边的参数等等，具体见其定义。状态机系统是一个比较精妙的系统，请务必仔细观看。

`calculate_weighted_offset_angle` 也是一个极其重要的函数，负责提取中线信息并锁定两个预瞄点，再将预瞄点转换为角度。之前我们使用了权重数组求方向，但是后来发现添加一个多达十多个参数的权重数组会导致权重极其难以调节，而且同时会严重引入震荡，所以我们后来就修改成了两个预瞄点的方式。

同时如果需要自己修改这个函数的话，处理过程中如果要加权，也就是我们惯性导航分支里面的加权版本函数的类似做法，请听从建议，一定要对中线数组进行等距定点数采样，固定采样点数，否则计算出来的偏转角度会与赛道边线长度挂钩。

常用搭配：

- 搭配 `my_task_function`：`tracking()` 中调用 `image_proc()`。
- 搭配 `my_global`：最终方向 `onto` 会给控制线程使用。
- 搭配 `my_image_transmitter`：调试时可以把图像和边线发给上位机。

### `project/code/red_rect.hpp` / `project/code/red_rect.cpp`

这是我们的裁框函数，负责从当前视角中裁剪一个固定大小的框，将结果转递给推理模型。

它不是盲裁，而是通过红色色块定位目标图片，再按模型输入尺寸缩放。它比较适合“目标图片下方贴红色定位条”的数据集和赛道布置方式。如果现场红条颜色、宽度、光照变化太大，需要优先改 `r_threshold`、`RED_ROW_RATIO`、有效区域这些参数。

常用搭配：

- 搭配 `tflm_model_process_lq`：裁出的 `target_roi` 直接送入模型推理。
- 搭配 `imgproc` 或主循环任务：从摄像头帧中拿图像进行检测。

### `project/code/tflm_model_process_lq.hpp` / `project/code/tflm_model_process_lq.cpp`

推理文件。众所周知，逐飞在他们的工程里面静态连接了由腾讯开源的这一基于龙芯框架优化过的 CPU 神经网络推理架构文件，但是我们就是不知道为什么他们不用。所以我们参考龙邱工程写了 `project/code/tflm_model_process_lq.cpp` 来实现推理，效率比之前逐飞使用的 TensorFlow 高了至少五倍。

命名乃版本遗留问题，懒得改了，不要问为什么命名成现在这个样子。龙邱模型如何使用请参考龙邱开源资料，当前这个文件是支持直接运行龙邱的 `.param` / `.bin` 的，当然我们的训练脚本之后也会开源出来。

常用搭配：

- 搭配 `red_rect`：先裁图，再推理。
- 搭配 `libraries/zf_components/ncnn`：推理底层依赖 ncnn。

### `project/code/my_image_transmitter.hpp` / `project/code/my_image_transmitter.cpp`

这是图像上位机传输模块，用 TCP 把 RGB565 图像发给逐飞助手相关接口。它主要用于调试：你可以把摄像头原图、处理后的图、带边线的图发到上位机看。

需要注意 `SERVER_IP` 和 `SERVER_PORT` 要和上位机一致。还有一点，RGB565 传输过程中有字节序交换，如果你看到上位机颜色很怪，别急着怀疑摄像头，先检查这里。

常用搭配：

- 搭配 `my_task_function`：`send_picture_to_Serve()` 会调用图像传输。
- 搭配 `imgproc`：调视觉时可以把边线、中线一起发出去。

## 构建文件

### `project/user/CMakeLists.txt`

这是工程构建入口，负责包含交叉编译工具链、寻找 OpenCV 和 ncnn、加入 `user`、`code` 和 `libraries` 下的源码，最后生成可执行文件。

它当前使用 `aux_source_directory` 自动收集源码，所以你在 `project/code` 下新增 `.cpp` 后一般不用手动加到构建列表里。但是新增第三方库、改 OpenCV 路径、改交叉编译工具链时，还是要回来看这个文件。至于 `USE_TTF` 字体开关，上面已经放到菜单栏目讲了，因为它最终影响的是菜单和屏幕显示效果。

常用搭配：

- 搭配 `project/user/cross.cmake`：交叉编译工具链配置。
- 搭配 `project/user/main.cpp`：最终程序入口在 user 目录。
- 搭配 `libraries`：逐飞底层库和 ncnn 静态库都从这里进入工程。

## 其他提醒

1. 控制类参数不要迷信默认值，换车模或者换安装方式后一定要重新标定。
2. 图像处理的中线数组如果要加权，尽量先等距采样再算，不然参数会和边线长度耦合。
3. 菜单功能页一定要实现返回，不然菜单会卡在功能页面。
4. IMU 初始化零漂统计时，小车必须静止，不然 yaw 角后面大概率会开始玄学。
5. 新增参数时同步改配置文件读取、写入、打印、上位机命令和控制器写入函数，不要只改一半。
